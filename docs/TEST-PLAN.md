# Тест-план первого запуска

Все тесты — от администратора. Начинать строго с `persistent: false`
(мягкий режим): любая проблема решается `sc stop VPNGuard`, фильтры
снимет сама ОС. Рядом держать `vpnguard panic` — на всякий случай.

## Фаза 0 — сборка

1. Заменить `YOURNAME` в go.mod и импортах, пуш → Actions зелёный.
   Ожидаемые места сюрпризов: два `NOTE(first-build)` (API tailscale/wf).
2. Закоммитить go.mod + go.sum из артефакта `go-mod-pinned`,
   включить кэш в workflow.

## Фаза 1 — мягкий режим, киллсвитч без VPN

3. `vpnguard init` → в конфиге: `persistent: false`, `enabled: true`,
   пути к openvpn.exe и .ovpn. `vpnguard service install`.
4. `vpnguard status` → ACTIVE, N правил.
   Лог: `C:\ProgramData\VPNGuard\vpnguard.log`.
5. VPN НЕ подключён. Проверки:
   - `ping 8.8.8.8` → **таймаут** (блок работает);
   - `nslookup example.com` → **отвечает** (dns_when_down: svchost);
   - браузер → **ничего не открывается**;
   - `ping 127.0.0.1` → ок (loopback).
6. `sc stop VPNGuard` → через пару секунд `ping 8.8.8.8` → **работает**
   (мягкий режим снял фильтры сам). `sc start VPNGuard` → снова блок.

## Фаза 2 — VPN и туннель

7. Подключить VPN через OpenVPN GUI. В логе службы: `adapters=[...]`.
   Проверки: браузер работает, IP на 2ip.ru/ifconfig.me — **IP VPN-сервера**.
8. Разорвать VPN (отключить в GUI). Немедленно: браузер мёртв,
   `ping 8.8.8.8` — таймаут. Переподключить — всё вернулось.
9. Жёсткий разрыв: убить openvpn.exe в диспетчере задач при активном
   скачивании чего-нибудь. Скачивание должно **встать мгновенно**
   (правило привязано к интерфейсу, не к событию).
10. Реконнект по имени: если в .ovpn hostname — переподключение из GUI
    должно пройти (svchost-DNS). Затем поставить `dns_when_down: off`,
    reload, повторить — должно пройти **по кэшу**
    (`resolve-cache.json` уже создан фазой 7).

## Фаза 3 — трей

11. `scripts\create-tray-task.bat` → иконка появилась, UAC не спрашивал.
    Цвета: зелёный при VPN, жёлтый после разрыва, серый после
    выключения киллсвитча тумблером, красный после `sc stop VPNGuard`.
12. Тумблер выкл → в config.yaml `enabled: false` (проверить файлом),
    сеть открыта. Тумблер вкл → блок вернулся.
13. Формы: добавить приложение в белый список, включить allowlist →
    при VPN ходит только оно (браузер не из списка — мёртв, но имена
    резолвит); выключить обратно.

## Фаза 4 — туннели

14. Добавить в «Туннели...» ssh-скрипт с health-check 127.0.0.1:1080.
    После подключения VPN — статус «работает» в меню трея, SOCKS отвечает.
15. Убить ssh.exe вручную → в меню «перезапуск», затем снова «работает».
16. Сломать health-check (закрыть порт на сервере) → после fail_threshold
    провалов процесс перезапущен, пауза растёт ×2 (видно в detail).
17. Разорвать VPN → туннели остановлены; подключить → поднялись сами.

## Фаза 5 — железный режим

18. `persistent: true`, reload. `sc stop VPNGuard` → блок **остался**
    (`ping 8.8.8.8` — таймаут, `vpnguard status` → ACTIVE).
19. Перезагрузка машины → после входа блок на месте до/без службы.
20. `vpnguard panic` → сеть открыта. `sc start VPNGuard` → блок вернулся.

Если все 20 пунктов прошли — киллсвитч железный и можно жить.
