# VPNGuard

Железный (fail-closed) kill switch для VPN под Windows на базе Windows
Filtering Platform (WFP) + супервизор ssh-туннелей (этап 3).

## Как это работает

- В ядро Windows через WFP устанавливается собственный sublayer с
  максимальным приоритетом и **persistent**-правилами: они переживают
  падение службы, убийство процесса и перезагрузку.
- По умолчанию **всё заблокировано**. Разрешены только: loopback, DHCP,
  NDP (IPv6-служебка), IP-адреса VPN-сервера (только для openvpn.exe),
  весь трафик через туннельный интерфейс, и — пока VPN не подключён —
  DNS по политике `dns_when_down` (по умолчанию только службе DNS
  Windows/svchost; можно `off`: реконнект пойдёт по кэшу IP).
- Два режима: `persistent: false` (мягкий — фильтры снимаются ОС при
  остановке/падении службы, окирпичиться невозможно; для обкатки) и
  `persistent: true` (железный — фильтры переживают всё).
- Служба следит за туннельным адаптером: адаптер поднялся — открылось
  «окно» через туннель; адаптер упал — окно закрылось *само*, потому что
  разрешение привязано к интерфейсу. Никакой гонки «заметили падение →
  успели заблокировать»: блок стоит всегда.

## Установка

Скачай `VPNGuard-Setup-vX.Y.Z.exe` из [Releases](../../releases) и запусти.
Инсталлер сам: создаст конфиг, откроет его в блокноте (укажи пути к
`openvpn.exe` и своему `.ovpn`, сохрани, закрой блокнот), установит и
запустит службу, настроит автозапуск трея с правами администратора без
UAC и покажет иконку в трее. Всё — дальше просто пользуйся OpenVPN GUI
как обычно.

Обновление — запустить новый Setup поверх (конфиг не трогается).
Удаление — стандартно через «Приложения»: деинсталлятор останавливает
службу и **снимает все фильтры из WFP**, сеть остаётся открытой.

Для тех, кто не любит инсталлеры, в релизах лежит
`VPNGuard-portable-win-x64-*.zip` — ручная установка описана в
docs/TEST-PLAN.md (vpnguard init → config → vpnguard service install).

## ⚠️ Аварийный выход

Kill switch — persistent. Если что-то пошло не так и сети нет:

```
vpnguard panic
```

(от администратора; полностью убирает все правила VPNGuard из WFP).
Команда работает даже если служба мертва или удалена. Держи её в голове
**до** того, как ставить службу.

Деинсталляция службы (`vpnguard service uninstall`) **не** снимает
правила — это осознанно. Снимать — только `vpnguard disable`.

## Заметки

- `app_policy: allowlist` — параноидальный режим: даже через туннель в
  сеть могут ходить только перечисленные .exe.
- Резолв хостнеймов VPN-сервера кэшируется в `resolve-cache.json`, чтобы
  служба могла восстановить правила при старте, когда DNS ещё заблокирован.
- Лог: `C:\ProgramData\VPNGuard\vpnguard.log`.
- Перед изменением go.mod замени `YOURNAME` на свой GitHub-аккаунт.

## Туннели

Секция `tunnels:` в конфиге (формат совместим со старым VPNSentinel).
Служба запускает их после поднятия VPN, гоняет TCP health-check по
локальному проброшенному порту, перезапускает с экспоненциальным
backoff (стартовая пауза ×2 до 60с, сброс после минуты стабильной
работы). Процессы живут в Job Object — при рестарте гарантированно
умирает всё дерево, включая сирот от .bat-скриптов.

Для ssh обязательно добавляйте в команду:
`-N -o ServerAliveInterval=10 -o ServerAliveCountMax=3 -o ExitOnForwardFailure=yes`
— без этого ssh может висеть «живым трупом» и держать мёртвый туннель.

## Дорожная карта

1. ✅ WFP kill switch + служба + CLI + CI
2. ✅ Монитор VPN: адаптер (авторитетный сигнал) + опциональный
   management interface OpenVPN для деталей статуса
3. ✅ Супервизор туннелей (healthcheck, backoff, job objects)
4. ✅ IPC: named pipe `\\.\pipe\VpnSentinelService`, протокол построчного
   JSON, совместимый с WinForms-треем VpnSentinel (команды: ping,
   status, enable, disable, reload) + горячий reload конфига
5. ✅ Трей (tray/VpnGuard.Tray, WinForms): статус, тумблер киллсвитча,
   формы настроек/приложений/туннелей поверх config.yaml + reload,
   panic из меню, автозапуск без UAC через планировщик
6. Инсталлятор (Inno Setup), WireGuard, кроссплатформенность

## Трей (этап 5 — готов)

`tray/VpnGuard.Tray` — WinForms-иконка в трее (C#, .NET 8): статус
(зелёный/жёлтый/серый/красный щит), вкл/выкл киллсвитча, формы
«Разрешённые приложения», «Туннели», «Настройки» (пишут config.yaml и
шлют reload службе), открытие конфига/лога, аварийный panic и
автозапуск без UAC через планировщик (`scripts\create-tray-task.bat`
или пункт меню). Общается со службой по named pipe — протокол описан в
`internal/ipc`. Собирается тем же workflow; релизный zip содержит оба
exe.

## Портативный режим и логи

Начиная с этой сборки все рабочие файлы лежат **рядом с exe**, а не в
ProgramData: `config.yaml`, `vpnguard.log`, `resolve-cache.json`. Кинул
папку с бинарником на флешку — конфиг поехал вместе с ним. Если папка с
exe только для чтения (например `C:\Program Files\...` после установки
инсталлером) — автоматически используется `%ProgramData%\VPNGuard`.
Переопределить явно: переменная окружения `VPNGUARD_DIR`.

Лог теперь подробный: при `vpnguard service run` он идёт и в консоль, и
в файл одновременно. В нём видно каждый шаг — какой endpoint распознан
из .ovpn, что разрешено и что заблокировано (каждое правило WFP:
`+ PERMIT w=100 vpn-endpoint ... [out-v4]`), события подключения VPN,
старт/стоп/health-check туннелей, все команды из трея. Первым делом при
любом глюке смотри `vpnguard.log` рядом с exe.
